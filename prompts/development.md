### 关键约束
1. **仓库和分支**
   - 所有改动必须在分支 {{branch}} 上进行，不能在 master 上直接提交
   - 开发前先 `git checkout {{branch}}`，若分支不存在则创建
   - 不要修改 git 配置或删除分支

2. **实现范围和方式**
   - 严格按照方案中的"涉及文件"和"实现步骤"执行
   - 不要超出方案范围添加额外功能
   - 代码风格和命名规范与现有代码保持一致

3. **数据库迁移**
   - 如果方案包含数据库变更，需要编写对应的迁移脚本
   - 迁移脚本放在 `backend/migrations/` 目录
   - 命名格式：`XXX_描述.sql`（例如 `034_add_archived_at.sql`）

4. **测试**
   - 开发完成后运行相关的单元测试或集成测试
   - 测试必须通过才能提交代码审查
   - 在方案中提及的关键功能点都应该有相应的测试覆盖

5. **代码注释和文档**
   - 新增复杂逻辑需要注释说明
   - 如果修改了 API 接口，需要更新接口文档（如有）
   - 不要提交大量代码却缺少说明

6. **Git 提交**
   - 不要提交 `git commit`，代码审查通过后由审查者或 CI 负责 commit
   - 只需确保改动已保存在工作目录和暂存区
